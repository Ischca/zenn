## はじめに

「毎回同じ説明をしている気がする」と感じたことはないでしょうか。
スキルはそういった繰り返しを減らすための仕組みです。
ただし、何でもスキル化すればいいというものでもありません。
この記事では、スキルの設計と使いどころを整理します。

## スキルとは

Claude Codeにおける「スキル」は、Claudeの能力を拡張するための仕組みです。

[公式ドキュメント](https://code.claude.com/docs/en/skills)では次のように説明されています。

> スキルはClaudeができることを拡張します。指示を含むSKILL.mdファイルを作成すると、Claudeはそれをツールキットに追加します。Claudeは関連するときにスキルを使用するか、`/skill-name`で直接呼び出すこともできます。

スキルは`.claude/skills/`ディレクトリにMarkdownファイルとして配置します。
Claudeは状況に応じてスキルを自動で読み込むか、ユーザーが`/skill-name`で明示的に呼び出すこともできます。

## CLAUDE.mdとスキルの違い

CLAUDE.mdとスキルは似ていますが、役割が異なります。

| 項目 | CLAUDE.md | スキル |
|------|-----------|--------|
| 読み込み | 常に自動 | 必要なときだけ |
| 内容 | コーディング規約、ビルドコマンド、「Xをするな」ルール | APIドキュメント、ワークフロー（deploy、reviewなど） |
| 目安 | 500行以下 | 必要に応じて分割 |

CLAUDE.mdはプロジェクトの基本情報、スキルは特定の知識やワークフローという違いがあります。

## スキルの種類

スキルには2つのタイプがあります。

[公式ドキュメント](https://code.claude.com/docs/en/skills)では次のように説明されています。

> **参照コンテンツ**：Claudeが現在の作業に適用する知識を追加します。規約、パターン、スタイルガイド、ドメイン知識。このコンテンツはインラインで実行され、Claudeは会話コンテキストと一緒に使用できます。
>
> **タスクコンテンツ**：特定のアクションに対するステップバイステップの指示をClaudeに与えます。デプロイ、コミット、コード生成など。これらは`/skill-name`で直接呼び出すことが多いアクションです。`disable-model-invocation: true`を追加すると、Claudeが自動でトリガーすることを防げます。

### 参照コンテンツの例（API設計パターン）
```markdown
---
name: api-conventions
description: このコードベースのAPI設計パターン
---

APIエンドポイントを書くときは以下に従う。
- RESTfulな命名規則を使う
- 一貫したエラーフォーマットを返す
- リクエストバリデーションを含める
```

### タスクコンテンツの例（Issue修正ワークフロー）
```markdown
---
name: fix-issue
description: GitHubのIssueを修正する
disable-model-invocation: true
---

GitHubのIssue: $ARGUMENTS を分析して修正する。

1. `gh issue view`でIssue詳細を取得
2. 問題を理解する
3. 関連ファイルを検索
4. 修正を実装
5. テストを書いて実行
6. lint/型チェックを通す
7. コミットしてPR作成
```

## なぜスキルが必要か

### コンテキストの効率的な利用

コンテキストウィンドウには上限があります。
全ての情報をCLAUDE.mdに詰め込むと、肥大化してパフォーマンスが低下します。

スキルを使えば、必要なときだけ情報を読み込ませることができます。
「API設計」の作業をしているときだけAPI規約のスキルが読み込まれ、「デプロイ」のときだけデプロイ手順のスキルが読み込まれる、という形です。

### 再利用性と品質の安定

同じ指示を毎回手で入力すると、表現のブレが生じます。
スキルとして定義しておけば、常に同じ品質の指示が渡されます。
また、`.claude/skills/`をバージョン管理に含めれば、チーム全体で共有できます。

## スキルの作成方法

### ファイル構成

```
.claude/skills/
├── review/
│   └── SKILL.md
└── deploy.md
```

ディレクトリ形式（`review/SKILL.md`）でも単一ファイル形式（`deploy.md`）でも作成できます。

### SKILL.mdの構造

```markdown
---
name: skill-name
description: このスキルがいつ使われるべきかの説明
allowed-tools: Read, Grep, Glob  # オプション
disable-model-invocation: true   # オプション
---

ここにClaudeへの指示を書く。
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| `name` | Yes | 一意の識別子。`/name`で呼び出せる |
| `description` | Yes | Claudeが自動読み込みを判断する際に使用 |
| `allowed-tools` | No | 使用可能なツールを制限 |
| `disable-model-invocation` | No | trueにするとClaudeが自動で呼び出さない |

### 保存場所

| 場所 | スコープ |
|------|----------|
| `.claude/skills/`（プロジェクト） | 現在のプロジェクトのみ |
| `~/.claude/skills/`（ユーザー） | 全プロジェクト |

## 実践的なスキル例

### セルフレビュー
```markdown
---
name: self-review
description: 実装後にコードを自己レビューする。コミット前に使用。
---

変更されたファイルを`git diff`で確認し、以下の観点でレビューする。
- 命名規則の遵守
- エラーハンドリングの適切さ
- テストの網羅性
- セキュリティ上の問題

問題があれば指摘し、修正案を提示する。
```

### ドキュメント更新
```markdown
---
name: update-docs
description: コード変更に伴うドキュメント更新
disable-model-invocation: true
---

以下のドキュメントを確認し、必要に応じて更新する。
1. README.md
2. API.md
3. CHANGELOG.md
```

## スキルを作るタイミング

同じ指示を何度も書いている、特定の作業で出力品質がばらつく、チームで共有したいパターンがある。
こういった状況になったら、スキル化を検討します。
逆に、一度きりの作業や、まだパターンが見えていない作業は、スキル化を急ぐ必要はありません。

## まとめ

- スキルはClaudeの能力を拡張する仕組み
- CLAUDE.mdは常に読み込まれ、スキルは必要なときだけ読み込まれる
- 参照コンテンツ（知識）とタスクコンテンツ（ワークフロー）の2種類がある
- `.claude/skills/`に配置し、バージョン管理でチーム共有可能
- 同じ指示を繰り返しているならスキル化を検討する
