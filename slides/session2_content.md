## Slide 04: Claude Codeは何を見ているか

**タイプ**: コンテンツ / **ラベル**: PART 1 — コンテキストとCLAUDE.md

**コンテキストウィンドウ = Claudeの作業メモリ**

メモリバーのイメージ:
```
[CLAUDE.md] [会話履歴] [読んだファイル] [ツール結果] [    空き    ]
|<------------- 約200,000トークン（日本語で約10万字）------------>|
```

- セッション開始時にCLAUDE.mdが自動で読み込まれる
- 会話が進むにつれ、コンテキストが蓄積していく
- ファイルの読み取り、コマンドの実行結果もコンテキストに入る

→ コンテキストウィンドウ = 「Claude Codeが今覚えていること」の総量

### 台本

Claude Codeが何を見ているのか。「コンテキストウィンドウ」という概念がまず大事です。
これはClaudeの作業メモリのようなものです。上限は約200,000トークン、日本語で約10万字程度。

セッションが始まると、まずCLAUDE.mdが自動で読み込まれます。そこに会話のやりとり、読んだファイル、コマンドの実行結果が積み重なっていきます。

このメモリには限りがある、ということを覚えておいてください。

## Slide 05: なぜ「忘れる」のか

**タイプ**: コンテンツ / **ラベル**: PART 1 — コンテキストとCLAUDE.md

**コンパクト: コンテキストの自動圧縮**

Before（コンテキストが上限に近づいたとき）:
```
[CLAUDE.md] [指示A] [ファイル読み] [指示B] [実装] [テスト] [修正]...
```

After（コンパクト発動後）:
```
[CLAUDE.md] [要約: 古い会話] [最近のやりとり]
```

**起きること**:
- 最初の指示や制約が要約・削除される場合がある
- 「このファイルは触らないで」と言ったのに忘れられる
- 長いセッションほどリスクが高い

**対策**:
- 重要な情報はCLAUDE.mdに書く（毎回読み込まれるので消えない）
- 作業が変わるタイミングで `/clear` する
- 長い作業はタスクを分割する

### 台本

Claude Codeが「忘れる」原因がこれです。コンパクトという自動圧縮が発生します。
コンテキストが上限に近づくと、古い情報が要約・削除されます。

なので、最初に「このファイルは触らないで」と言っても、長いセッションの後では忘れられることがある。

対策として一番重要なのが、重要な情報はCLAUDE.mdに書くこと。CLAUDE.mdは毎セッション最初に読み込まれるので、コンパクトで消えない「永続メモリ」として機能します。

作業が変わるタイミングで `/clear` でリセットするのも有効です。

## Slide 06: CLAUDE.mdとは

**タイプ**: コンテンツ / **ラベル**: PART 1 — コンテキストとCLAUDE.md

**人間のオンボーディング資料と同じ発想**

新しいメンバーが来たら、何を教えるか:
- このプロジェクトはこういう構成です
- ビルドはこのコマンドで
- こういうルールがあるので守ってください
- ここは触らないでください

→ これをそのままCLAUDE.mdに書く

**配置場所と読み込みルール**:

| 場所 | 読み込み | 用途 |
|---|---|---|
| `~/CLAUDE.md` | 常に | 個人の全プロジェクト共通設定 |
| `プロジェクト/CLAUDE.md` | 常に | プロジェクト固有の設定（チーム共有） |
| `サブディレクトリ/CLAUDE.md` | そのディレクトリで作業時 | モジュール固有の注意事項 |

→ チームでバージョン管理すれば、全員が同じ「教え方」でClaude Codeを使える

### 台本

CLAUDE.mdは、人間のオンボーディング資料と同じ発想です。新しいメンバーに渡す引き継ぎドキュメントと同じように、プロジェクトのルールや構成を書きます。

配置場所が3つあります。ホームディレクトリ、プロジェクトルート、サブディレクトリ。

プロジェクトルートのCLAUDE.mdが一番よく使います。これをgitで管理すれば、チーム全体で同じ設定を共有できます。

## Slide 07: CLAUDE.mdに書くべき内容

**タイプ**: コンテンツ / **ラベル**: PART 1 — コンテキストとCLAUDE.md

**4カテゴリに分けて考える**:

**1. プロジェクト概要**
```markdown
## リポジトリ概要
Zennに公開する技術記事を管理するリポジトリ。
```

**2. よく使うコマンド**
```markdown
## コマンド
npm run preview      # ローカルプレビュー
npm run lint         # textlintで記事をチェック
npm run lint:fix     # textlintで自動修正
```

**3. コーディング規約・ルール**
```markdown
## Python
Python を使う場合は必ず `uv` を使う（pip は使わない）。
```

**4. やってはいけないこと**
```markdown
## 禁止事項
- main ブランチに直接プッシュしない
- .env ファイルを編集しない
```

→ PRでよく指摘される項目 = CLAUDE.mdに書くべき内容

### 台本

CLAUDE.mdに何を書くかは、4つのカテゴリで整理するとわかりやすいです。
プロジェクト概要、よく使うコマンド、コーディング規約、そしてやってはいけないこと。

特に「PRでよく指摘される項目」は書いておくべきです。暗黙知を言語化する良い機会にもなります。

ここに出している例は、この勉強会資料を管理しているリポジトリの実物です。

## Slide 08: CLAUDE.mdの始め方と育て方

**タイプ**: コンテンツ / **ラベル**: PART 1 — コンテキストとCLAUDE.md

**/init で自動生成 → 育てる → 刈り込む**

**Step 1: /init で自動生成**
```
> /init
# プロジェクトの技術スタック、ディレクトリ構成を自動検出
# CLAUDE.md のたたき台が生成される
```

**Step 2: チーム固有の情報を追記**
- PRでよく指摘される項目
- 触ってはいけないファイル
- チーム固有のコーディング規約

**Step 3: 肥大化を防ぐ**
- 目安は500行以下
- 詳細な仕様は別ファイルに分け、必要なときだけ参照させる
- 不要になった情報は定期的に削除

**Tips**: `/insights` で改善提案を受ける
- 過去のセッションから「CLAUDE.mdに追加すべき内容」を提案してくれる

### 台本

まずは `/init` で自動生成するのが一番早いです。プロジェクトの技術スタックやディレクトリ構成を自動検出してくれます。

そのたたき台にチーム固有の情報を追記していきます。

大事なのが、CLAUDE.mdは育てていくものだということ。最初から完璧を目指さない。使いながら「あ、これも書いておこう」と追記していく。

ただし肥大化には注意。500行を超えると、パフォーマンスが低下する可能性があります。モデルが従える指示数には上限があるためです。

ちなみに `/insights` というコマンドで、過去のセッションから改善提案をもらえます。具体的な文章まで出してくれるので、そのままコピーして貼り付けられます。

## Slide 14: スキルの2タイプと実例

**タイプ**: コンテンツ / **ラベル**: PART 2 — スラッシュコマンド & Skills

**参照コンテンツ — 知識を渡す**

```markdown
---
name: api-conventions
description: APIエンドポイント作成時の設計パターン
---

APIエンドポイントを書くときは以下に従う。
- RESTfulな命名規則を使う
- 一貫したエラーフォーマットを返す
- リクエストバリデーションを含める
```

→ Claudeが関連する作業を検知したとき、自動で読み込まれる

**タスクコンテンツ — 手順を渡す**

```markdown
---
name: fix-issue
description: GitHubのIssueを修正する
disable-model-invocation: true
---

GitHubのIssue: $ARGUMENTS を分析して修正する。
1. gh issue viewでIssue詳細を取得
2. 関連ファイルを検索
3. 修正を実装
4. テストを書いて実行
5. コミットしてPR作成
```

→ `/fix-issue 42` のように明示的に呼び出す

### 台本

スキルには2つのタイプがあります。参照コンテンツとタスクコンテンツです。

参照コンテンツは「知識」。API設計パターンのように、関連する作業をClaudeが検知したときに自動で読み込まれます。

タスクコンテンツは「手順」。`/fix-issue 42` のように明示的に呼び出して使います。
`disable-model-invocation: true` を設定すると、自動では呼び出されず明示的に使うときだけ読み込まれます。

下の例では `$ARGUMENTS` で引数を受け取っています。Issueの番号を渡すと、そのIssueの修正ワークフロー全体が自動で回ります。

## Slide 17: Hooksとは

**タイプ**: コンテンツ / **ラベル**: PART 3 — Hooks

**イベント駆動の自動化**

Claude Codeの特定のタイミングで、自動でコマンドを実行する仕組み

```
[ファイル編集] --PostToolUse--> [lint --fix を自動実行]
[コミット]     --PreToolUse---> [テストを自動実行]
```

**設定場所**: `.claude/settings.json`

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": "npm run lint --fix" }]
      }
    ]
  }
}
```

**主なイベント**:

| イベント | タイミング | 使い道 |
|---|---|---|
| `PreToolUse` | ツール実行前 | 禁止操作のブロック |
| `PostToolUse` | ツール実行後 | lint / formatter の自動実行 |
| `Notification` | 通知時 | 外部への通知連携 |
| `Stop` | セッション終了時 | レポート生成 |

→ CIのpre-commitフックに近い概念。Claude Codeの操作に品質ゲートを挟む

### 台本

Hooksは、Claude Codeの特定のタイミングで自動でコマンドを実行する仕組みです。
CIのpre-commitフックを知っている方は、あれに近い概念です。

たとえば、ファイル編集後に自動でlintを走らせる。コミット前にテストを自動実行する。

設定は `.claude/settings.json` に書きます。これもgit管理すれば、チーム全体で同じ品質ゲートを共有できます。

エージェントの出力品質は一定ではありません。同じ指示でも結果がばらつくことがある。Hooksで機械的にチェックを挟むことで、品質を担保できます。

## Slide 18: Hooksの実践例

**タイプ**: コンテンツ / **ラベル**: PART 3 — Hooks

**例1: 編集後にlintを自動修正**
```json
{
  "matcher": "Edit|Write",
  "hooks": [{ "type": "command", "command": "npm run lint --fix $CLAUDE_FILE_PATH" }]
}
```
Claudeがファイルを編集するたび、lint違反が自動修正される

**例2: 危険なコマンドをブロック**
```json
{
  "matcher": "Bash",
  "hooks": [{
    "type": "command",
    "command": "echo $CLAUDE_TOOL_INPUT | grep -q 'rm -rf' && exit 1 || exit 0"
  }]
}
```
`rm -rf` を含むコマンドの実行を事前にブロック

**例3: Goファイル編集後に型チェック**
```json
{
  "matcher": "Edit|Write",
  "hooks": [{ "type": "command", "command": "go vet ./..." }]
}
```

**ポイント**:
- Hooksが失敗するとClaudeに通知される → Claudeが修正を試みる
- 「lint実行して」と毎回指示する必要がなくなる
- チーム全員が同じチェックを自動実行

### 台本

具体例を3つ紹介します。

例1: 編集後のlint自動修正。Claudeがファイルを編集するたびlintが走って自動修正される。

例2: 危険なコマンドのブロック。`rm -rf` を含むBashコマンドの実行を事前にブロックできます。PreToolUseに設定します。

例3: Goプロジェクトなら `go vet` を編集後に走らせるなど、言語に合わせた設定ができます。

重要なポイントとして、Hooksが失敗するとClaudeにその結果が通知されます。するとClaudeが自分で修正を試みてくれます。自動チェックとエージェントの自己修正が組み合わさるのが強力なところです。
