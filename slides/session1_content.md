## Slide 03: まずは調査だけ任せてみる

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**よくある痛み**: 調査・影響範囲の把握で時間が溶ける

**入口**: 最初は実装ではなく「調査・要約・差分整理」だけ

→ いきなり実装させず、業務の一部から始める

### 台本

いきなり実装を任せる必要はありません。読み取り専用で始められるのでリスクも低い。
Before/Afterは体感の話なので、人やリポジトリによって差はあります。

## Slide 04: Claude Code でできること

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

- ターミナルで動くAIエージェント。指示は自然言語でOK
- 調査 → 計画 → 実装 → 検証 → 取りまとめ、を一連で扱う
- デフォルトは読み取り専用。ファイル編集やコマンド実行は許可制
- コード補完ではなく「作業を進める」ツール

**右側**: フロー図（調査 → 計画 → 実装 → 検証 → 要約）

### 台本

Claude Codeは、ターミナルで動くAIエージェントです。
「AIにコードを書かせるツールでしょ？」と思われがちですが、コード生成はできることの一部にすぎません。
調査、計画、実装、検証、取りまとめ。これを一つのセッションで連続的に扱えます。
安全面としては、デフォルトは読み取り専用で、ファイル編集やコマンド実行には都度許可が要ります。勝手に何かを壊す心配はありません。

## Slide 05: Claude Code のツール群

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**基本ツール（上段・コンパクト）**:

| ツール | できること |
|---|---|
| Read / Grep / Glob | ファイルの閲覧・コード検索 |
| Edit / Write | ファイルの編集・作成 |
| WebSearch / WebFetch | Web検索・ページ取得 |
| Task | サブエージェントによる並列探索 |

**Bash — あらゆるコマンドを実行できる（下段・大きく）**:

```
say "完了しました"           # 音声で通知
marp slides.md --pptx        # スライド生成
ffmpeg -i video.mp4 out.gif  # 動画をGIFに変換
git log --oneline -10        # 直近のコミットを確認
npm test                     # テスト実行
```

→ 開発作業に限らない。コマンドで実現できることは、何でも

### 台本

Claude Codeが使えるツールを見てみます。
基本ツールとして、ファイルの読み書き・検索、Web検索、サブエージェントがあります。これらはClaude Codeの内蔵ツールです。
注目はBash。文字通り「あらゆるコマンド」を実行できます。
テストやビルドはもちろん、marpでスライドを生成する、ffmpegで動画を変換する。こういったことも全部Bashの守備範囲です。
コマンドで実現できることは、原理的にすべてClaude Codeに任せられます。
どのツールを使うかは指定不要で、指示に応じてClaude Codeが自動で組み合わせます。
ちなみにスライドではgitを例に出していますが、Perforceなら `p4 changes -m 10` で同じようなことができます。コマンドラインから操作できるツールなら何でも使えます。

## Slide 06: ChatGPT・Copilotとの使い分け

**タイプ**: コンテンツ

| ツール | 得意領域 | ひとことで |
|---|---|---|
| ChatGPT | 設計議論・仕様の言語化 | 壁打ち |
| GitHub Copilot | IDE内のコード補完 | 書く |
| Claude Code | リポジトリ全体の作業フロー | 進める |

→ 排他的ではなく補完的。併用するのがおすすめ

### 台本

この3つのツールは競合ではなく、補完的な関係です。得意領域が違います。
ChatGPTは壁打ちに強い。設計議論、仕様の言語化、アプローチの比較に向いています。
Copilotはコードを書くことに強い。IDEから離れずに入力を補助してくれます。
Claude Codeは一連の作業フローを進めることに強い。調査から実装、差分整理まで通しで扱えます。
大事なのは「どれか1つ」ではなく、使い分けることです。

## Slide 07: 開発フローがどう変わるか

**タイプ**: コンテンツ

**左列 — 従来の開発フロー（すべて自分）**

1. コードを読む
2. 方針を考える
3. コードを書く
4. テスト実行
5. デバッグ・修正
6. 差分整理・PR作成

**右列 — Claude Code を使った開発フロー**

1. 目標・背景を伝える（Developer）
2. コード調査（Claude）
3. 計画提案（Claude） → レビュー（Developer）
4. 実装（Claude）
5. テスト・検証（Claude） → 確認（Developer）
6. 差分整理・PR作成（Claude）

→ 開発者の出番は「方向づけ」と「判断」の2回だけになる

### 台本

従来の開発では、全工程を自分で回します。6ステップすべてが自分の作業です。
Claude Codeを使うと、開発者の出番は2箇所に絞られます。最初の方向づけと、途中の判断。
これは「サボる」ではなく「判断に集中する」ということです。
「コード補完」と「エージェント」の本質的な違いはここにあります。

## Slide 08: 具体例で見てみる

**タイプ**: コンテンツ

**サブタイトル**: 例：「CIが落ちた。原因を調査して修正案を出して」

**横4列カード（タイムライン形式）**:

| Developer | Claude | Developer | Claude |
|---|---|---|---|
| **方向づけ** | **調査 → 計画提案** | **判断** | **実行 → 取りまとめ** |

### 台本

開発者の介入は2回。方向づけと判断だけです。
「丸投げ」ではありません。要所で判断を入れるから品質を保てます。
原因調査に30分かかるところが、5分で俯瞰できるイメージです。

## Slide 10: チャットボットとエージェントの違い

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**左 — チャットボット**（電話サポートに近い）
- 質問 → 回答 → 質問 → 回答
- 1回のやりとりが完結
- ユーザーが手順を考え、1ステップずつ指示する

**右 — エージェント（Claude Code）**（仕事を任せた同僚に近い）
- ゴールを伝えたら自律的に実行
- ファイル読み書き・コマンド実行・Web検索
- 調査 → 計画 → 実装を一連で判断・実行する

→ 「1行ずつ指示する」のではなく「ゴールと背景を伝えて任せる」

### 台本

チャットボットとエージェントは根本的に違います。
チャットボットは電話サポートに近い。こちらが1ステップずつ指示して、その都度答えが返ってきます。
エージェントは仕事を任せた同僚に近い。ゴールと背景を伝えたら、やり方は本人が判断します。
同僚に仕事を任せるとき、1行ずつ「次はこのファイルを開いて、次はこの関数を見て」とは指示しませんよね。
「CIが落ちてるから原因を調べて」と伝えて、やり方は任せる。
この「委任」の感覚が、Claude Codeではとても大事です。

## Slide 11: 指示の出し方

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**効果的な指示の4要素**:

1. **目標** — 何を達成したいか
2. **背景・制約** — なぜ必要か / やってはいけないこと
3. **現状** — 今どうなっているか
4. **期待する出力** — どういう形式で結果がほしいか

#### Bad

```
この機能を改善して
```

#### Good

```
CIが落ちている。
現状：昨日のリファクタ以降失敗するようになった。
目標：ログを要約し、原因候補を3つ挙げて。
調査結果は箇条書きでまとめて。
コードの変更はまだしないこと。
```

### 台本

指示の4要素。目標、背景・制約、現状、期待する出力です。
全部揃っている必要はありません。ただし、目標は必須です。
悪い例を見てください。「この機能を改善して」。何を？どう？がまったくわかりません。
良い例は、さっきと同じCIのシナリオです。現状・目標・制約・出力形式が明確になっています。
「コードの変更はまだしないこと」が地味に大事で、やってほしくないことを書くと暴走を防げます。
指示が具体的であるほど、結果も具体的になります。

## Slide 12: たたき台から始める

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**たたき台 → 改善ループ**:

1. 指示を出す → 2. 出力を確認 → 3. フィードバック → 4. 改善版

**3つのテクニック**:

| テクニック | 具体例 |
|---|---|
| 「なぜ」を聞く | 期待と違ったら、修正指示の前に理由を聞く。自分の指示の問題が見えることがある |
| 提案形式で伝える | 「Xをして」より「Xを考えているが、どう思うか」。別の選択肢が出てくる |
| 手直ししたら伝える | 自分で編集した場合はClaudeに共有する。伝えないと古い状態で動き続ける |

→ 「全然違う」より「この部分は良い、ここを変えて」

### 台本

最初から完璧を求めない。これが一番大事なマインドセットです。
まず出力を見て、良い点と悪い点を具体的にフィードバックして、改善版を出してもらいます。
テクニックを3つ紹介します。
1つ目：「なぜそうしたのか」と聞く。自分の指示に問題があったケースも多いです。
2つ目：命令ではなく提案で伝える。「Xを考えているけど、どう思う？」と聞くと、自分では気づかなかった選択肢が出てきます。
3つ目：自分でコードを直接編集したら、必ずClaudeに伝えてください。伝えないと古い状態を前提に動いてしまいます。

## Slide 14: Plan Mode とは

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

**Plan Mode = 読み取り専用のモード**

- ファイル閲覧・検索・Web検索は可能
- ファイル編集・コマンド実行は不可
- 調査と計画立案に集中するためのモード

**なぜ計画から始めるのか**:

- 計画段階で認識のズレに気づける
- コードを壊すリスクがゼロ
- 手戻りを最小限にしてから実装に入れる

**操作方法**:
- `Shift+Tab` で切替（Normal → Auto Accept → Plan）
- `claude --permission-mode plan` で起動

### 台本

Plan Modeは読み取り専用のモードです。ファイルを見たり検索はできますが、編集やコマンド実行はできません。
なぜ計画から始めるのか。公式のベストプラクティスにも「計画ステップがないと、Claudeは直接コーディングに飛び込む傾向がある」と書かれています。
計画なしで実装を始めると、前提の認識ズレで根本から作り直しになることがあります。
Plan Modeなら壊すリスクはゼロです。安心して調査・計画できます。
操作はShift+Tabで切替。起動時に --permission-mode plan で指定もできます。
補足ですが、Extended Thinkingとは別の概念です。Extended Thinkingは深く考えさせる機能。Plan Modeはツールの制限。両方を同時に使うこともできます。

## Slide 15: Plan Mode の実践ワークフロー

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| Step | やること | 伝えること |
|---|---|---|
| Step 1 | Plan Modeで調査・計画 | 「計画を立てて。実装はまだしないで。」 |
| Step 2 | 計画をレビュー | 「後方互換性はどうする？」「テストは？」 |
| Step 3 | 実装を許可 | 「OK、実装して」（調査だけならここで完了） |
| Step 4 | 結果を確認 | テスト実行 → PR作成 |

**Step 1で伝えておくと良いこと**:
- 受け入れ基準（どうなったら完了か）
- 禁止事項（絶対にやってはいけない変更）
- 想定外の扱い（計画にない問題を見つけたらどうするか）

→ 小さな修正でもこの流れを習慣にすると手戻りが減る

### 台本

4ステップのワークフローです。
Step 1：「計画を立てて。実装はまだしないで。」この一言が鍵です。
このとき、受け入れ基準や禁止事項も伝えておくとブレません。たとえば「既存APIは壊さない」「計画にない問題を見つけたら報告して止まって」など。
Step 2：計画が返ってきたらレビューします。「後方互換性は？」「テストは？」と聞いてみてください。
Step 3：計画に納得したら「OK、実装して」。調査だけが目的ならここで完了です。
Step 4：テストを実行して、結果を確認してコミット。
小さな修正でもこの流れを習慣にすると、手戻りが劇的に減ります。
