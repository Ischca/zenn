## Slide 03: まずは調査だけ任せてみる

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**よくある痛み**: 調査・影響範囲の把握で時間が溶ける

**Before / After**:

- **Before**: チケット1件の調査に30分。ログを追い、参照箇所を探し、影響範囲を整理する
- **After**: Claude Codeに任せると数分で返ってくる。調査結果・原因候補・影響範囲がまとまる

**入口**: 最初は実装ではなく「調査・要約・差分整理」だけ

→ いきなり実装させず、業務の一部から始める

### 台本

調査・影響範囲の把握は時間が溶けやすい。ここが最初の狙いどころ。
いきなり実装を任せる必要はない。読み取り専用で始められるのでリスクも低い。
Before/Afterは体感の話。人やリポジトリによって差はある。
ただ「調査の時間が減る」のは共通して体感しやすいポイント。
この感覚を持ってもらった上で、もう少し詳しく見ていく。

## Slide 04: Claude Code でできること

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

- ターミナルで動くAIエージェント。指示は自然言語でOK
- 調査 → 計画 → 実装 → 検証 → 取りまとめ、を一連で扱う
- デフォルトは読み取り専用。ファイル編集やコマンド実行は許可制
- コード補完ではなく「作業を進める」ツール

**右側**: フロー図（調査 → 計画 → 実装 → 検証 → 要約）

### 台本

Claude Codeはターミナルベースのエージェント型開発支援ツール。
「AIにコードを書かせるツールでしょ？」と思われがちだが、コード生成はできることの一部。
調査、計画、実装、検証、取りまとめを一つのセッションで連続的に扱える。
安全面：デフォルトは読み取り専用で、ファイル編集やコマンド実行には都度許可が必要。勝手に壊す心配はない。

## Slide 05: ChatGPT・Copilotとの使い分け

**タイプ**: コンテンツ

| ツール | 得意領域 | ひとことで |
|---|---|---|
| ChatGPT | 設計議論・仕様の言語化 | 壁打ち |
| GitHub Copilot | IDE内のコード補完 | 書く |
| Claude Code | リポジトリ全体の作業フロー | 進める |

→ 排他的ではなく補完的。併用するのがおすすめ

### 台本

3つのツールは競合ではなく補完的。得意領域が異なる。
ChatGPTは壁打ちに強い。設計議論、仕様の言語化、アプローチ比較。
Copilotはコードを書くことに強い。IDEから離れずに入力を補助。
Claude Codeは一連の作業フローを進めることに強い。
IDEで記述し、ターミナル側で調査・実行・差分整理という分担が自然。
大事なのは「どれか1つ」ではなく使い分けること。

## Slide 06: 開発フローがどう変わるか

**タイプ**: コンテンツ

**左列 — 従来の開発フロー（すべて自分）**

1. コードを読む
2. 方針を考える
3. コードを書く
4. テスト実行
5. デバッグ・修正
6. 差分整理・PR作成

**右列 — Claude Code を使った開発フロー**

1. 目標・背景を伝える（Developer）
2. コード調査（Claude）
3. 計画提案（Claude） → レビュー（Developer）
4. 実装（Claude）
5. テスト・検証（Claude） → 確認（Developer）
6. 差分整理・PR作成（Claude）

→ 開発者の出番は「方向づけ」と「判断」の2回だけになる

### 台本

従来の開発では全工程を自分で回す。6ステップすべてが自分の作業。
Claude Codeを使うと、開発者の出番は2箇所に絞られる。最初の方向づけと、途中の判断。
これは「サボる」ではなく「判断に集中する」ということ。
「コード補完」と「エージェント」の本質的な違いはここにある。

## Slide 07: 具体例で見てみる

**タイプ**: コンテンツ

**サブタイトル**: 例：「CIが落ちた。原因を調査して修正案を出して」

**横4列カード（タイムライン形式）**:

| Developer | Claude | Developer | Claude |
|---|---|---|---|
| **方向づけ** | **調査 → 計画提案** | **判断** | **実行 → 取りまとめ** |
| "CIが落ちた。ログを要約して原因候補を3つ挙げて" | "失敗箇所はX。依存関係の不整合が濃厚。原因候補を3つ提示します" | "候補2を深掘りして。既存仕様は変えないで" | "根拠と修正案を整理。テストも追加済み。" |

→ 「丸投げ」ではなく「要所で判断する」。この感覚が重要

### 台本

具体的なタスクで流れを見てみる。
「CIが落ちた」という日常的なシナリオ。
開発者がやるのは、目標を伝えること（30秒）と、計画をレビューすること（1分）。
Claudeが調査・計画提案と結果の取りまとめを担当する。
開発者の介入は2回。方向づけと判断。
「丸投げ」ではない。要所で判断を入れるから品質を保てる。
原因調査に30分かかるところが、5分で俯瞰できるイメージ。
ここからPART 2で「伝え方のコツ」を見ていく。

## Slide 09: チャットボットとエージェントの違い

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**左 — チャットボット**（電話サポートに近い）
- 質問 → 回答 → 質問 → 回答
- 1回のやりとりが完結
- ユーザーが手順を考え、1ステップずつ指示する

**右 — エージェント（Claude Code）**（仕事を任せた同僚に近い）
- ゴールを伝えたら自律的に実行
- ファイル読み書き・コマンド実行・Web検索
- 調査 → 計画 → 実装を一連で判断・実行する

→ 「1行ずつ指示する」のではなく「ゴールと背景を伝えて任せる」

### 台本

チャットボットとエージェントは根本的に違う。
チャットボットは電話サポートに近い。こちらが1ステップずつ指示して、その都度答えが返ってくる。
エージェントは仕事を任せた同僚に近い。ゴールと背景を伝えたら、やり方は本人が判断する。
同僚に仕事を任せるとき、1行ずつ「次はこのファイルを開いて、次はこの関数を見て」と指示しない。
「CIが落ちてるから原因を調べて」と伝えて、やり方は任せる。
この「委任」の感覚が、Claude Codeではとても重要。

## Slide 10: 指示の出し方

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**効果的な指示の4要素**:

1. **目標** — 何を達成したいか
2. **背景・制約** — なぜ必要か / やってはいけないこと
3. **現状** — 今どうなっているか
4. **期待する出力** — どういう形式で結果がほしいか

#### Bad

```
この機能を改善して
```

#### Good

```
CIが落ちている。
現状：昨日のリファクタ以降失敗するようになった。
目標：ログを要約し、原因候補を3つ挙げて。
調査結果は箇条書きでまとめて。
コードの変更はまだしないこと。
```

### 台本

指示の4要素：目標、背景・制約、現状、期待する出力。
全部揃っている必要はない。しかし、目標は必須。
悪い例：「この機能を改善して」。何を？どう？がまったく不明。
良い例：Slide 07と同じCIのシナリオ。現状・目標・制約・出力形式が明確。
「コードの変更はまだしないこと」が地味に大事。やってほしくないことを書くと暴走を防げる。
ポイント：指示が具体的であるほど、結果も具体的になる。

## Slide 11: たたき台から始める

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**たたき台 → 改善ループ**:

1. 指示を出す → 2. 出力を確認 → 3. フィードバック → 4. 改善版

**3つのテクニック**:

| テクニック | 具体例 |
|---|---|
| 「なぜ」を聞く | 期待と違ったら、修正指示の前に理由を聞く。自分の指示の問題が見えることがある |
| 提案形式で伝える | 「Xをして」より「Xを考えているが、どう思うか」。別の選択肢が出てくる |
| 手直ししたら伝える | 自分で編集した場合はClaudeに共有する。伝えないと古い状態で動き続ける |

→ 「全然違う」より「この部分は良い、ここを変えて」

### 台本

最初から完璧を求めない。これが一番大事なマインドセット。
まず出力を見て、良い点と悪い点を具体的にフィードバックし、改善版を出してもらう。
テクニック3つ。
1つ目：「なぜそうしたのか」と聞く。自分の指示に問題があったケースも多い。
2つ目：命令ではなく提案。「Xを考えているが、どう思うか」と聞くと、自分では気づかなかった選択肢が出てくる。
3つ目：自分でコードを直接編集したら、必ずClaudeに伝える。伝えないと古い状態を前提に動いてしまう。

## Slide 13: Plan Mode とは

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

**Plan Mode = 読み取り専用のモード**

- ファイル閲覧・検索・Web検索は可能
- ファイル編集・コマンド実行は不可
- 調査と計画立案に集中するためのモード

**なぜ計画から始めるのか**:

- 計画段階で認識のズレに気づける
- コードを壊すリスクがゼロ
- 手戻りを最小限にしてから実装に入れる

**操作方法**:
- `Shift+Tab` で切替（Normal → Auto Accept → Plan）
- `claude --permission-mode plan` で起動

### 台本

Plan Modeは読み取り専用。ファイルを見たり検索はできるが、編集やコマンド実行はできない。
なぜ計画から始めるのか。Anthropicのベストプラクティスにも明記されている。
「ステップ1-2は重要。これがないと、Claudeは直接コーディングに飛び込む傾向がある」と。
計画なしで実装を始めると、前提の認識ズレで根本から作り直しになることがある。
Plan Modeなら壊すリスクはゼロ。安心して調査・計画できる。
操作はShift+Tabで切替。起動時に --permission-mode plan で指定もできる。
Extended Thinkingとは別の概念。
Extended Thinking = 深く考えさせる（think, think hard）。Plan Mode = ツール制限。
両者の併用も可能。

## Slide 14: Plan Mode の実践ワークフロー

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| Step | やること | 伝えること |
|---|---|---|
| Step 1 | Plan Modeで調査・計画 | 「計画を立てて。実装はまだしないで。」 |
| Step 2 | 計画をレビュー | 「後方互換性はどうする？」「テストは？」 |
| Step 3 | 実装を許可 | 「OK、実装して」（調査だけならここで完了） |
| Step 4 | 結果を確認 | テスト実行 → PR作成 |

→ 小さな修正でもこの流れを習慣にすると手戻りが減る

### 台本

4ステップのワークフロー。
Step 1：「計画を立てて。実装はまだしないで。」この一言が鍵。
Step 2：計画が返ってきたらレビューする。「後方互換性は？」「テストは？」と聞く。
Step 3：計画に納得したら「OK、実装して」。調査だけが目的ならここで完了。
Step 4：テストを実行し、結果を確認してコミット。
小さな修正でもこの流れを習慣にすると手戻りが劇的に減る。

## Slide 15: 計画段階のチェックリスト

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| 項目 | 確認すること |
|---|---|
| ゴール | 何を達成したいか |
| 背景・制約 | なぜ必要か / やってはいけないこと |
| 受け入れ基準 | どうなったら完了か |
| 禁止事項 | 絶対にやってはいけない変更 |
| 想定外の扱い | 計画外の問題が見つかったらどうするか |

→ これが曖昧なまま実装に入ると、後から認識のズレが表面化する

### 台本

計画段階で確認すべき5つの項目。
ゴール：何を達成したいか。最も重要。
背景・制約：なぜ必要か。技術的な制約も含む。
受け入れ基準：どうなったら完了か。「テストが通る」「レビューOK」など。
禁止事項：「既存APIの破壊はダメ」「特定ファイルは変えないで」など。
想定外の扱い：計画にない問題を見つけたら「報告して止まる」か「自己判断で対処」か。
Slide 10の4要素は「1回の指示」の話。こちらは「プロジェクト全体の計画」のスコープ。
