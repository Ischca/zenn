## Slide 04: Claude Code とは

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**ターミナルで動くAIエージェント**
自然言語で指示するだけで、コードベース全体を読み・調べ・作業を進める

1. **自然言語で指示** — 日本語で伝えるだけで動く。コマンドの知識は不要
2. **プロジェクト全体を把握** — ファイルを横断的に読み、関連箇所を自分で探す
3. **勝手には動かない** — ファイル編集やコマンド実行は都度あなたの許可が必要

→ コード補完ではなく「作業を丸ごと進める」新しいタイプのツール

### 台本

まず「Claude Codeとは何か」を一言で説明すると、ターミナルで動くAIエージェントです。

自然言語で指示することができ、プロジェクト全体を見渡せ、そして勝手には動かない、というのが基本的な部分になってます。

これだけだと分かりづらい
どういうものなのかイメージをお見せします。

## Slide 08: Claude Code のツール群

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**主なビルトインツール**:

| ツール | できること |
|---|---|
| Read / Grep / Glob | ファイルの閲覧・コード検索 |
| Edit / Write | ファイルの編集・作成 |
| WebSearch / WebFetch | Web検索・ページ取得 |
| Task | サブエージェントによる並列探索 |

**Bash — あらゆるコマンドを実行できる**:

```
say "完了しました"           # 音声で通知
marp slides.md --pptx        # スライド生成
ffmpeg -i video.mp4 out.gif  # 動画をGIFに変換
git log --oneline -10        # 直近のコミットを確認
ical-buddy add "MTG 15:00"   # カレンダーに予定を追加
```

→ 開発作業に限らない。ターミナルで実現できることは、何でも

### 台本

Claude Codeが使えるツールについてです。
ここで言う「ツール」とは、Claudeが自分の判断で呼び出せる機能のことです。

基本ツールとして、ファイル検索や読み書き・、Web検索、Taskというものがあります。これらはClaude Codeの内蔵ツールです。

ユーザーが指示を出すと、Claudeがどのツールを、どの順番で使うか、自分で考えて実行します。
なので、「ファイルを読んで」「検索して」と細かく一つずつ指示する必要はありません。

特にBashツールが強力。ターミナル上で使える「あらゆるコマンド」を実行できます。
テストやビルドなどもそうですが、スライドを生成したり、動画を変換したり。
カレンダーに予定を追加する、なんてこともそういうコマンドがあればできてしまいます。
スライドではgitの例が出てますが、Perforceなら p4 コマンド で同じようなことができます。

という感じで、コードを書く人だけでなく、日常業務の自動化にも使えるものではあります。

「何でも実行できる」と聞くと不安に思うかもしれないですが、
Claude Codeはファイル編集やコマンド実行のたびに、必ずユーザーの許可を求めます。
勝手に何かを実行して壊すということないですが、あまり確認せず許可をしてしまうと危険なので、注意しましょう。

## Slide 09: ChatGPT・Copilotとの使い分け

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

| | ChatGPT | GitHub Copilot | Claude Code |
|---|---|---|---|
| どこで使う？ | ブラウザのチャット画面 | VSCodeの中（エディタ拡張） | ターミナル（エディタに依存しない） |
| 何をしてくれる？ | 質問すると答えてくれる | コードの続きを提案してくれる | 指示すると理解→調査→実行→確認まで一連で動く |
| やりとりの例 | 「この設計どう思う？」 | （書いてると候補が出る） | 「サーバーでエラー出た。ログから原因調べて」 |
| 見ているもの | あなたが貼ったテキストだけ | 今開いているファイル | プロジェクトのファイル全部＋Webも検索 |
| あなたの役割 | 質問を考える・回答を判断する | 補完候補をTab/Escで選ぶ | 指示・判断・確認 |

→ 排他的ではなく補完的。併用するのがおすすめ

### 台本

ChatGPTやCopilotとの比較・使い分けについてさらっと

基本的に、この3つのツールは競合するのではなく、補完的な関係にあります。


ChatGPTは壁打ちに強いと思っています。DeepSearchなどもありますし。
そういった設計議論、仕様の言語化、アプローチの比較に向いています。


Copilotはコードを自前で書くときに便利。Claude Codeで全てのコードを書かせるわけにはいかないので、Copilotの補助もあるといいです。

Claude Codeはコードベースを見ながら、一連の作業フローを進めることに強い。調査から実装、差分整理まで通しでできます。
コードについて質問したいこととかも、Claude Codeを使うといいと思います。

というように、大事なのは「どれか1つ」ではなく、使い分けることです。

## Slide 10: 開発フローがどう変わるか

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**左列 — 従来の開発フロー（すべてDeveloper）**

1. コードを読む
2. 方針を考える
3. コードを書く
4. テスト実行
5. デバッグ・修正
6. 差分整理・PR作成

**右列 — Claude Code を使った開発フロー**

1. 目標・背景を伝える（Developer）
2. コード調査（Claude）
3. 計画提案（Claude） → レビュー（Developer）
4. 実装・テスト・検証（Claude）
5. 差分整理・PR作成（Claude）

→ 開発者の出番は「指示」「判断」「確認」。要所で人間が関わる

### 台本

開発フローがどう変わるかについてす。
従来の開発では、全工程を自分で回します。6ステップすべてが自分の作業です。

Claude Codeを使うと、開発者の出番は3箇所に絞られます。最初の指示、途中の判断、最後の確認という感じです。

大半のことをClaude Codeがやってくれるので、結果として人間の作業は「指示」、「判断」、「確認」の比重が増えます。

## Slide 11: 具体例で見てみる

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

**サブタイトル**: 例：「サーバーでエラーが出た。ログから原因を調べて」

**横5列カード（タイムライン形式）**:

| Developer | Claude | Developer | Claude | Developer |
|---|---|---|---|---|
| **指示** | **調査 → 報告** | **判断** | **実行 → 完了報告** | **確認** |
| サーバーでエラーが出た。どういうエラーか確認して --- (GCPのログを貼り付け) --- | APIのタイムアウトエラーです。直近のデプロイで接続先の設定が変わったのが原因です | 設定を戻す方向で修正して。テストも作成して。 | 設定を修正してテストを追加しました。全テスト通過しています | 差分を確認してマージ |

→ 丸投げではなく、要所で人間が関わる

### 台本

実際の使い方のイメージです。

「サーバーでエラーが出た」という場面。GCPのログを貼り付けて「まずどういうエラーか確認して」と伝える。これが最初の指示です。

ポイントは、形式ばった指示を出さなくていいこと。Slackで同僚に仕事を依頼するような言い方でOKです。

するとClaudeがログを読み、コードの差分を確認し、原因を特定して報告してきます。

その報告を見て「設定を戻す方向で」と判断を返す。ここで人が方向性を決めています。

Claudeが修正とテスト追加まで行い、完了を報告します。

そして最後に、開発者が差分を確認してマージする。この確認ステップが品質を担保します。

開発者の出番は「指示」「判断」「確認」の3回。調査や実装の実作業はClaudeが担いますが、要所で必ず人間が関わります。
これがClaude Codeとの付き合い方の基本です。

これでPART 1は以上です。
なんとなく、どんなもので、どう動くのか、がつかめたと思います。

## Slide 13: チャットボット vs エージェント

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**左 — チャットボット**（電話サポートに近い）
- 質問 → 回答 → 質問 → 回答
- 1回のやりとりが完結
- プロンプトテンプレートが有効

**右 — エージェント（Claude Code）**（仕事を任せた同僚に近い）
- 指示して、まず動いてもらう
- 出力を見て判断・修正する
- 対話の中で方向を修正していく

→ チャットボットの作法をそのままエージェントに持ち込まない

### 台本

まず、チャットボットとエージェントは違う

チャットボットは1ステップずつ指示して、その都度答えが返ってきます。
なのでプロンプトテンプレートのような「こう聞けばうまくいく」という型が有効です。

一方、エージェントは仕事を任せた同僚に近い。ゴールと背景を伝えたら、やり方は本人が判断します。
ここで大事なのは、巨大な1プロンプトで全部を完璧に指示しようとしないことです。
それをすれば勝手に色々動いてはくれますが、動いているうちにコンテキストが切迫して、最初の指示を徐々に忘れていってしまいます。
また、「やっていくうちに諸々別の問題が発覚し、当初の計画通りにいかないことがわかる」ということは、人間ではよくありますがエージェントでも同様です。

エージェントの場合は、まず指示して動いてもらい、出てきたものを見て判断・修正しながら進めるのが正しい使い方です。
チャットボットで有効だったプロンプトテンプレート的なアプローチを、そのままエージェントに持ち込むとうまくいきません。
コミュニケーションの設計自体が違うということがわかると、性能を引き出せるようになります。

## Slide 14: 初手の指示

じゃあ、最初の指示では何を伝えればいいのか。
背景・現状、そして方向性です。

ポイントは、これは「完璧なプロンプトを書く」必要は無い

例は先ほどと似たような話で、こう言う問題が出てますという、現状。直近のデプロイ以降といった背景情報。

## Slide 15: 対話しながら進める

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**ループ**: 1. 方向付け → 2. たたき台を見る → 3. 議論・修正 → 4. 次の一手

**対話のコツ**:

| コツ | 説明 |
|---|---|
| 「なぜ」を聞く | 期待と違ったら、すぐ修正指示を出さず理由を聞く。自分の方向付けの問題が見えることも |
| 提案形式で議論する | 「Xして」ではなく「Xを考えているが、どう思うか」。別の選択肢が出てくる |
| 手直ししたら共有する | 自分で編集した場合は必ず伝える。伝えないと古い状態で動き続ける |

### 台本

エージェントとの仕事は対話で進めるのが基本です。


最初の指示以降、何度かディスカッションをするということがよくあります。
そのときの対話のコツを3つ紹介します。


1つ目：「なぜそうしたのか」と聞く。期待と違う出力が出たとき、すぐ修正指示を出すのではなく、まず理由を聞いてみる。自分の方向付けに問題がある場合も多いです。

2つ目：命令ではなく提案形式で議論する。「Xして」ではなく「Xを考えているけど、どう思う？」と。これも人と話すような感じです。

3つ目：自分でコードを直接編集したら、必ずClaudeに共有してください。伝えないと古い状態を前提に動いてしまいます。これも人と同じで、情報共有が大事です。

## Slide 17: Plan Mode とは

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

**概要**:
- 読み取り専用のモード
- ファイル閲覧・検索・Web検索は可能
- ファイル編集・コマンド実行は不可
- 調査と計画立案に集中するためのモード

**操作方法**:
- `Shift+Tab` で切替（Normal → Auto Accept → Plan）
- `claude --permission-mode plan` で起動

**なぜ計画から始めるのか**:
- 認識のズレに早く気づける — 計画段階で認識のズレに気づける
- コードを壊すリスクがゼロ — 読み取り専用なので安全に試行錯誤できる
- 手戻りを最小限にできる — 手戻りを最小限にしてから実装に入れる

### 台本

Plan Modeは読み取り専用のモードです。
ファイルを見たり検索はできますが、編集やコマンド実行はできません。

なぜ計画から始めるのか。
計画なしで実装を始めると、最初の認識ズレでどんどん変な方向に行ってしまい結果やり直しだったり、不具合が隠れていたりとかが起こります。

Plan Modeは読み取り専用なので、安全に調査・計画できます。

操作はShift+Tabで切替。起動時に --permission-mode plan で指定もできます。

## Slide 18: Plan Mode の実践ワークフロー

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| Step | やること | 伝えること | 説明 |
|---|---|---|---|
| Step 1 | Plan Modeで調査・計画 | 「こういう計画を立てて。こういうところに懸念があるよ」 | Claudeがコードを読み、方針を提案する |
| Step 2 | 計画をレビュー | 「後方互換性はどうする？」「テストは？」 | 不明点をつぶし、計画を確定する |
| Step 3 | 実装を許可 | 「OK、実装して」/ 調査だけならここで完了 | 計画に沿って実装、または調査結果を活用 |
| Step 4 | 結果を確認 | テスト実行 → PR作成 | 変更内容を取りまとめて完了 |

**Step 1で伝えておくと良いこと**:
- 受け入れ基準（どうなったら完了か）
- 禁止事項（絶対にやってはいけない変更）
- 想定外の扱い（計画にない問題を見つけたらどうするか）

### 台本

Plan Mode を使うなら。

Step 1：「計画を立てて。こういうところに懸念があるよ」この一言が鍵です。
計画モードでは、受け入れ基準や禁止事項も伝えておくといいです。一気にがーっと調査をするので。なるべく前提知識を伝えましょう。

Step 2：計画が返ってきたらレビューします。「後方互換性はどうなってる？」「テストはどうする？」と聞いてみてください。

Step 3：問題無さそうなら「OK、実装して」。調査だけがここで完了です。

Step 4：テストを実行したり、結果を確認してコミット。

小さな修正でもこの流れを習慣にすると、手戻りが劇的に減ります。
