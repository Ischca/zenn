## Slide 04: Claude Code とは

**タイプ**: コンテンツ / **ラベル**: PART 1 — Claude Code とは

- ターミナル上で動くAIエージェント
- 調査 → 修正 → 検証 → 取りまとめを一連で扱う
- ファイルの読み書き、コマンド実行を自律的に行う（都度許可 or 自動承認を選べる）
- デフォルトは読み取り専用。編集やコマンド実行には明示的な許可が必要
- 「コードを書く」ではなく「作業を進める」ツール

**右側**: フロー図（調査 → 修正 → 検証 → 要約）

### 台本

Claude Codeはターミナルベースのエージェント型開発支援ツール。
GitHub Copilotのような単発のコード補完とは違い、
調査・修正・検証・変更内容の取りまとめを、
ひとつのセッション内で連続して扱える。
安全面：デフォルトは読み取り専用で、ファイル編集やコマンド実行には都度許可が必要。勝手に何か壊す心配はない。
デモGIFがあれば見せる。

## Slide 06: 開発フローの変化

**タイプ**: コンテンツ

**左列 — 通常の開発フロー（すべて Developer）**

1. コードを読む
2. 方針を考える
3. コードを書く
4. テスト実行
5. デバッグ・修正
6. 差分整理・PR作成

**右列 — Claude Code を使った開発フロー**

1. 目標・背景を伝える（Developer）
2. コード調査（Claude）
3. 計画提案（Claude） → レビュー（Developer）
4. 実装（Claude）
5. テスト・検証（Claude） → 確認（Developer）
6. 差分整理・PR作成（Claude）

→ 開発者の役割が「実行者」から「方向づけ + レビュー」に変わる  
→ 介入ポイントは2回だけ（方向づけ / 判断）

### 台本

通常の開発では全工程を自分で回す。
Claude Codeを使うと、開発者の仕事は「何をしたいか伝える」と「要所でレビューする」に変わる。
ターミナルだけの話ではない。設計→実装→検証のワークフロー全体が変わる。
これが「コード補完」と「エージェント」の本質的な違い。
次のスライドで、具体的なタスクを例に、この流れを見てみる。

### 台本

## Slide 09: チャットボット vs エージェント

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**左 — チャットボット**
- 質問 → 回答 → 質問 → 回答
- ピンポン型のやりとり
- 1回のやりとりが完結
- ユーザーが手順を管理する

**右 — エージェント（Claude Code）**
- 目標を委任 → 自律的に実行
- ファイル読み書き・コマンド実行
- 調査 → 計画 → 実装を一連で行う
- エージェントが手順を判断する

→ コミュニケーションの方法も変わる。「指示」ではなく「委任」の意識が重要

### 台本

チャットボットとエージェントの根本的な違い。
チャットボット：ユーザーが1ステップずつ指示する。受動的。
エージェント：ゴールを伝えたら、自分で探索・計画・実行する。自律的。
アナロジー：チャットボットは電話サポート、エージェントは仕事を任せた同僚。
同僚に仕事を任せるとき、1行ずつ何をするか指示しない。
ゴールと背景を伝えて、やり方は任せる。
この「委任」の感覚がClaude Codeでは重要。

## Slide 10: 効果的な指示の4要素

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

1. **目標** — 何を達成したいか
2. **背景・制約** — なぜ必要か / やってはいけないこと
3. **現状** — 今どうなっているか
4. **期待する出力** — どういう形式で結果がほしいか

**テンプレ**: 目標 / 背景・制約 / 現状 / 期待する出力

#### Bad

```
この機能を改善して
```

#### Good

```
CIでtests/authのテストが3件落ちている。
現状：昨日のリファクタ以降失敗するようになった。
目標：テストを全件通す。
原因を調査して、修正案を提案して。
既存の振る舞いは変えないこと。
```

### 台本

指示の4要素：目標、背景・制約、現状、期待する出力。
全部揃っている必要はないが、目標は必須。
悪い例：「この機能を改善して」→ 何を改善？どう改善？が不明。
良い例：現状・目標・制約が明確。
ポイント：指示が具体的であるほど、結果も具体的になる。

## Slide 11: コンテキストの渡し方

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

| 方法 | 説明 | 例 |
|---|---|---|
| 画像ペースト | スクショやデザインを共有 | エラー画面のキャプチャを貼る |
| URL指定 | Webページを読ませる | `このドキュメントを読んで: https://...` |
| ファイル名を伝える | 対象ファイルを明示する | `src/auth.ts を見て` |
| ディレクトリ指定 | Claudeに探索させる | このディレクトリを調べて |
| パイプ（上級者向け） | コマンド出力を渡す | `cat error.log \| claude -p "原因を教えて"` |

※ VS Code拡張では `@ファイル名` でファイルを参照できる

### 台本

コンテキストを渡す方法はいくつかある。
画像やURLなど、非CLIでも渡せる方法から始めるのが安全。
ファイル名を伝えて対象を明示する方法は基本的。VS Code拡張なら@記法も使える。
画像もコピー＆ペーストで渡せる。エラー画面やデザインカンプなど。
URLを渡してWebページを読ませることもできる。
パイプでコマンド出力を渡す方法は上級者向け。ログ分析などに便利。-p フラグでプロンプトも一緒に渡せる。
ディレクトリ指定でClaude自身に探索させることも可能。

## Slide 12: たたき台 → 改善ループ

**タイプ**: コンテンツ / **ラベル**: PART 2 — コミュニケーション

**ループ図**: 1. 指示を出す → 2. 出力を確認する → 3. 具体的にフィードバック → 4. 改善版を受け取る

**実践テクニック**:

| テクニック | 説明 |
|---|---|
| 「なぜ」を聞く | 期待と違う動きをしたら、すぐ修正を指示する前に理由を聞く |
| 提案形式で伝える | 「Xをして」より「Xを考えているが、どう思うか」 |
| 手直ししたら伝える | 自分で編集した場合はClaudeに共有する |
| 初回で完璧を求めない | 「全然違う」より「この部分は良い、ここを変えて」 |

### 台本

最初から完璧を求めず、たたき台→改善のループで進める。
テクニック：
- 「なぜそうしたのか」と聞くと、自分の指示の問題が見えることがある
- 提案形式で伝えると、Claudeが別の選択肢を出してくれる
- 自分で直接コードを編集したら、その旨を伝える。伝えないと古い状態を前提に動く
- 「全然違う」と言うより「この部分は良い、この部分は変えて」と具体的に

## Slide 14: Plan Mode とは

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

- 読み取り専用の実行モード
- ファイル閲覧・検索は可能。Webアクセスは権限設定次第
- ファイル編集・コマンド実行は不可
- 調査と計画立案に集中するためのモード

**ポイント**:

- まずPlan Modeで調査・計画を固める
- 実装はレビュー後に許可する
- 最初はPlan Mode固定にする

**操作方法**:

- セッション中の切替: `Shift+Tab` または `Alt+M`
- 起動時に指定: `claude --permission-mode plan`

### 台本

Plan Modeは読み取り専用のモード。
まずPlan Modeで方針を固め、実装はレビュー後に許可する。
最初はPlan Mode固定で始めると失敗しにくい。
切替はShift+TabかAlt+Mで。起動時に --permission-mode plan で指定もできる。
Extended Thinkingとは別の概念。
Extended Thinking = 深く考えさせる（think, think hard）
Plan Mode = 使えるツールを制限する
両者の併用も可能。

## Slide 15: 実践ワークフロー

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| Step | やること | 指示例 | 結果 |
|---|---|---|---|
| Step 1 | Plan Mode で調査・計画 | `「計画を立てて。実装はまだしないで。」` | Claudeがコードを読み、方針を提案する |
| Step 2 | 計画をレビュー | `「後方互換性はどうする？」「テストは？」` | 不明点をつぶし、計画を確定する |
| Step 3 | 実装を許可して進める | `「OK、実装して」` | 計画に沿って実装が進む |
| Step 4 | 結果を確認・コミット | `テスト実行 → PR作成` | 変更内容を取りまとめて完了 |

### 台本

計画 → レビュー → 実装 → 確認の4ステップ。
ポイント：
- 「まだコードを書かないで」と明示的に伝える
- 計画段階で疑問点をつぶす。「後方互換性は？」「テストは？」
- 計画に納得したらモード切替で実装を許可
- 小さな修正でもこの流れを習慣にすると手戻りが減る

## Slide 16: 計画段階のチェックリスト

**タイプ**: コンテンツ / **ラベル**: PART 3 — Plan Mode

| 項目 | 説明 |
|---|---|
| ゴール | 何を達成したいか |
| 背景・制約 | なぜ必要か / やってはいけないこと |
| 受け入れ基準 | どうなったら完了か |
| 禁止事項 | 絶対にやってはいけない変更 |
| 想定外の扱い | 計画外の問題が見つかったらどうするか |

→ これらが曖昧なまま実装に入ると、後から認識のズレが表面化する

### 台本

計画段階で確認すべき5つの項目。
ゴール：何を達成したいか。最も重要。
背景・制約：なぜ必要か。技術的な制約も含む。
受け入れ基準：どうなったら完了か。テストが通る、レビューOKなど。
禁止事項：既存APIの破壊、特定ファイルの変更禁止など。
想定外の扱い：計画外の問題は報告して止まる、自己判断で対処、など。

## Slide 17: まとめ

**タイプ**: コンテンツ

**3つの要点**:

1. **Claude Code は「作業を進める」ツール** — コード補完ではなく、調査→修正→検証→取りまとめの一連フロー
2. **伝え方が結果を左右する** — 目標・背景・現状・期待出力を明確にし、たたき台→改善ループで進める
3. **計画から始める習慣をつける** — Plan Mode で方針を固めてから実装に入ると手戻りが減る

### 台本

まとめの3つの要点を読み上げ。
今日の持ち帰りは「面倒な調査を、まず任せる」。
次回は「カスタマイズする」がテーマ。
コンテキスト管理、Skills、サブエージェント、Plan Modeの深掘りを扱う。
質疑応答の時間へ。
