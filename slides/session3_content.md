## Slide 05: スキルとは

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**スラッシュコマンドとして使える拡張機能**

Claude Codeには `/batch` `/simplify` 等の組み込みスキルがあり、自作スキルも `.claude/skills/` に置くだけで追加できる

```
.claude/skills/
├── review/
│   └── SKILL.md      # /review で呼び出せる
└── deploy.md          # /deploy で呼び出せる
```

**CLAUDE.mdとの違い**:

| | CLAUDE.md | スキル |
|---|---|---|
| 読み込み | 常に自動 | 必要なときだけ |
| 内容 | プロジェクトの基本ルール | 特定の作業手順・知識 |
| 目安 | 500行以下 | 必要に応じて分割 |
| 例 | コーディング規約、ビルドコマンド | レビュー手順、デプロイフロー |

→ **CLAUDE.md = 常識、スキル = 専門知識**
→ スキルはAgent Skills Open Standardに準拠。Copilot・Codex CLI等でも動作する

### 台本

スキルは、スラッシュコマンドとして呼び出せる拡張機能です。
`/batch` や `/simplify` のように最初から組み込まれているものと、`.claude/skills/` にMarkdownファイルを置いて自作するものがあります。今日は主にカスタムスキルの作り方を紹介しますが、最後に組み込みの `/batch` がworktreeと組み合わさるとどうなるか、という話もします。

前回学んだCLAUDE.mdとの違いが大事です。CLAUDE.mdは「常に」自動で読み込まれる「常識」。スキルは「必要なときだけ」読み込まれる「専門知識」です。

CLAUDE.mdに全部詰め込むとコンテキストを圧迫します。状況に応じて読み込ませたい情報はスキルに分けるのが良いです。

補足ですが、スキルはAgent Skills Open Standardという業界標準に準拠しています。Claude Code用に作ったスキルが、GitHub CopilotやOpenAI Codex CLIでもそのまま動きます。学習コストが一度で済むのも嬉しいポイントです。

## Slide 06: スキルの2タイプ

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**左: 参照コンテンツ — 知識を渡す**

```markdown
---
name: api-conventions
description: APIエンドポイント作成時の
             設計パターン
---

APIエンドポイントを書くときは以下に従う。
- RESTfulな命名規則を使う
- 一貫したエラーフォーマットを返す
- リクエストバリデーションを含める
```

→ Claudeが関連する作業を検知したとき、自動で読み込まれる

**右: タスクコンテンツ — 手順を渡す**

```markdown
---
name: issue-planner
description: Issueの調査と設計を行う
disable-model-invocation: true
---

Issue #$ARGUMENTS の設計を行う。
1. gh issue viewでIssue詳細を取得
2. 関連コードを検索し影響範囲を特定
3. 実装方針をまとめる
4. ISSUE.mdに設計メモを追記
```

→ `/issue-planner 42` のように明示的に呼び出す

### 台本

スキルには2つのタイプがあります。

参照コンテンツは「知識」。API設計パターンのように、関連する作業をClaudeが検知したときに自動で読み込まれます。

タスクコンテンツは「手順」。`/issue-planner 42` のように明示的に呼び出して使います。
`disable-model-invocation: true` を設定すると、自動では呼び出されず、スラッシュコマンドとして呼び出したときだけ読み込まれます。

`$ARGUMENTS` で引数を受け取れるので、Issue番号を渡すと調査から設計まで一気に進みます。
さらに `` !`command` `` 構文を使うと、シェルコマンドの実行結果をスキル本文に動的に注入できます。たとえば `` !`git branch --show-current` `` と書けば、現在のブランチ名がスキル実行時に展開されます。
たとえばissue-workerでworktreeを作った後、issue-plannerで設計を固めてから実装に入る、といった連携ができます。

## Slide 07: 実践的なスキル例

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**例1: Issue作業スキル**

```markdown
---
name: issue-worker
description: GitHub Issueを分析し、worktreeで作業開始まで導く
disable-model-invocation: true
---

$ARGUMENTS のIssue番号それぞれについて以下を実行する。
複数番号がスペース区切りで渡された場合はすべて処理する。

1. gh issue view で Issue 詳細を取得
2. git worktree add で作業ブランチを作成
3. Issue内容から ISSUE.md を自動生成
4. 新しいworktreeでClaude Codeを起動
```

使い方: `/issue-worker 42`

**例2: PR作成スキル**

```markdown
---
name: pr-creator
description: チームのテンプレートに沿ってPRを作成する
disable-model-invocation: true
---

PRを作成する。

1. git diff で変更内容を確認
2. 以下のテンプレートに沿って本文を作成
   - ## 概要（何をしたか）
   - ## 背景（なぜやるか）
   - ## 影響範囲
   - ## テスト結果
3. gh pr create で提出
```

使い方: `/pr-creator`

> **いつ作るか**: 同じ指示を何度も書いている → スキル化を検討
> `.claude/skills/` をgit管理すれば、チーム全員が同じスキルを使える

### 台本

チームで共有すると嬉しいスキル例を2つ紹介します。

1つ目は issue-worker。Issue番号を渡すだけで、worktreeの作成からISSUE.mdの生成まで自動で行います。この後のPART 3で紹介するworktreeとの組み合わせで、複数Issueの並列開発ができるようになります。

2つ目はPR作成。PRのフォーマットがバラバラだと、レビューする側も大変です。テンプレートをスキル化しておけば、チーム全体のPR品質が揃います。

どちらも `.claude/skills/` に置いてgit管理すれば、チーム全員が同じスキルを使えます。スキルを作るタイミングは、同じ指示を何度も手打ちしていると感じたとき。逆に一度きりの作業なら急いでスキル化する必要はありません。

## Slide 08: スキルの設計ポイント

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**設定フィールド**:

| フィールド | 必須 | 説明 |
|---|---|---|
| name | Yes | 一意の識別子。`/name` で呼び出せる |
| description | Yes | Claudeが自動読み込みを判断する際に使用 |
| allowed-tools | No | 使用可能なツールを制限（Read, Grepなど） |
| disable-model-invocation | No | trueにするとClaudeが自動で呼び出さない |
| argument-hint | No | 引数のヒント。`/name` の後に補完表示される |
| model | No | 実行時のモデルを指定（例: haiku） |
| context | No | `fork` でサブエージェントとして隔離実行 |

**保存場所**:

| 場所 | スコープ |
|---|---|
| `.claude/skills/`（プロジェクト） | 現在のプロジェクトのみ。git管理でチーム共有 |
| `~/.claude/skills/`（ユーザー） | 全プロジェクト共通。個人用 |

**設計のコツ**:
- descriptionが鍵。ここを読んでClaudeが自動読み込みを判断する
- スキルの中身はClaudeへの指示そのもの。人に頼むように書く
- 1スキル1責務。大きくなったら分割する

→ ここまでが「スキル = メイン会話の中で動く仕組み」。次は「独立して動く」サブエージェントへ

### 台本

スキルの設計で押さえておくべきポイントです。

`allowed-tools` で使えるツールを制限できます。たとえばレビュー用スキルなら読み取り系ツールだけに絞る、といった使い方です。`argument-hint` を設定すると、スラッシュコマンド入力時に引数のヒントが表示されて使いやすくなります。

`context: fork` を設定すると、スキルが独立したサブエージェントのコンテキストで実行されます。重い処理でもメインの会話を圧迫しません。先ほどのサブエージェントとの使い分けで言えば、スキルにサブエージェントの特性を持たせる設定です。

逆に `user-invocable: false` を設定すると、メニューには表示されず、Claudeが関連する作業を検知したときだけ自動で読み込まれる背景知識として機能します。

保存場所はプロジェクトとユーザーの2つ。チーム共有ならプロジェクト側に、個人用ならユーザー側に置きます。

スキルの作成には skill-creator というツールも用意されています。対話形式でスキルを作成し、eval（テスト）やベンチマークで品質を検証できます。スキルが増えてきたら活用してみてください。

ここまでがスキルの話です。スキルは「メイン会話の中で動く仕組み」。次に紹介するサブエージェントは「独立して動く仕組み」です。この違いが重要です。

## Slide 09: サブエージェントとは

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**独立したコンテキストで動く専門エージェント**

```
[メインエージェント]
  ├── 指示: 「セキュリティの問題がないか調べて」
  │
  ├── [サブエージェント]  ← 独立コンテキスト
  │     └── 数十ファイルを検索・分析
  │
  └── 結果: 要約だけが返る ← コンテキスト消費を抑える
```

**本質**: コンテキストの分離
- サブエージェントの作業結果の要約だけがメイン会話に返される
- メイン会話のコンテキスト消費を大幅に削減できる

**なぜ必要か**:
- 大量のファイルを検索する → メインのコンテキストがすぐ埋まる
- サブエージェントに委譲すれば → メインは要約だけ受け取る
- 結果、メインの作業に影響しない

### 台本

サブエージェントは、独立したコンテキストで動く専門エージェントです。

たとえば「セキュリティの問題がないか調べて」と指示すると、サブエージェントが数十のファイルを読んで調査しますが、メインの会話に返されるのは要約だけです。

前回学んだように、コンテキストは有限な資源です。大量のファイルを検索する作業をメインでやるとすぐにコンテキストが埋まりますが、サブエージェントに委譲すればメインのコンテキストを守れます。

## Slide 10: 組み込み & カスタムサブエージェント

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**3つの組み込みサブエージェント**:

| 名前 | モデル | 用途 |
|---|---|---|
| Explore | Haiku（高速） | コードベースの検索・分析（読み取り専用） |
| Plan | — | Planモード中の調査 |
| general-purpose | Sonnet | 複雑なマルチステップタスク（全ツール利用可） |

→ Claudeが状況に応じて自動的に使い分ける

**カスタムサブエージェント — `.claude/agents/` にMarkdownファイルを配置**:

```markdown
---
name: code-reviewer
description: コードレビューの専門家。
             コード変更後に積極的にレビュー。
tools: Read, Grep, Glob, Bash
model: sonnet
memory: project
---

あなたはシニアコードレビューアーです。

呼び出されたときは以下を実行する。
1. git diffで最近の変更を確認
2. 変更されたファイルに焦点を当てる
3. 即座にレビューを開始
```

→ descriptionが重要。Claudeはこの説明を見て委譲先を判断する
→ `isolation: worktree` を追加すると、サブエージェントが独立したworktreeで動作する

### 台本

組み込みのサブエージェントが3つあります。Exploreは高速な検索特化型、Planはプランモード中の調査用、general-purposeは汎用タイプです。これらはClaudeが自動的に使い分けてくれます。

カスタムのサブエージェントも作れます。`.claude/agents/` にMarkdownファイルを置くだけです。スキルと同じような配置方法です。

descriptionが重要で、「コード変更後に積極的にレビュー」と書いておけば、コード変更があったときに自動的にこのサブエージェントが呼ばれるようになります。

フロントマターに `isolation: worktree` を追加すると、サブエージェントが独立したworktreeで動きます。ファイルの競合を気にせず並列で動かせるので、大規模な変更に有効です。この詳細はPART 3で触れます。

`memory: project` を設定すると、サブエージェントがセッションをまたいで知見を蓄積します。レビューで見つけたパターンや頻出する問題を覚えてくれるので、使うほど賢くなります。スコープは `project`（チーム共有）、`user`（個人・全プロジェクト共通）、`local`（個人・プロジェクト固有）の3つです。

## Slide 11: スキル vs サブエージェント

**タイプ**: コンテンツ / **ラベル**: PART 1 — Skills & サブエージェント

**どちらもClaudeの能力を拡張するが、動作が異なる**

| | スキル | サブエージェント |
|---|---|---|
| コンテキスト | メイン会話内 | 独立（要約だけ返る） |
| 向いている作業 | 知識参照、軽量なワークフロー | 大量の探索、並列実行 |
| 配置場所 | `.claude/skills/` | `.claude/agents/` |
| 設定ファイル | SKILL.md | Markdown |

**使い分けの基準**:
- 1〜3ファイルの作業 → スキル
- 数十ファイルの探索 → サブエージェント
- 結果を会話で直接使いたい → スキル
- メインのコンテキストを守りたい → サブエージェント

→ **スキル = 手順書、サブエージェント = 専門チームメンバー**

### 台本

PART 1の締めくくりとして、スキルとサブエージェントの使い分けを整理します。

どちらもClaudeの能力を拡張しますが、動作が違います。

スキルはメイン会話の中で動くので結果がそのまま残ります。軽量な作業に向いています。

サブエージェントは独立コンテキストで動いて要約だけ返ります。重い探索作業に向いています。

覚え方として、スキルは「手順書」、サブエージェントは「専門チームメンバー」。
手順書は自分で見て実行する。チームメンバーには仕事を任せて報告だけもらう。

## Slide 13: Hooksとは

**タイプ**: コンテンツ / **ラベル**: PART 2 — Hooks & MCP

**Claude Codeの特定のタイミングで、自動で処理を実行する仕組み**

設定場所: `.claude/settings.json`

**例1: 編集後にlintを自動修正**（PostToolUse）
```json
{
  "matcher": "Edit|Write",
  "hooks": [{
    "type": "command",
    "command": "npm run lint:fix"
  }]
}
```

**例2: テストを自動実行**（PostToolUse）
```json
{
  "matcher": "Edit|Write",
  "hooks": [{
    "type": "command",
    "command": "npm test -- --bail"
  }]
}
```

**例3: 完了通知**（Notification）
```json
{
  "hooks": [{
    "type": "command",
    "command": "say '完了しました'"
  }]
}
```

**ポイント**: Hooksが失敗 → Claudeに通知 → Claudeが自動で修正を試みる

### 台本

Hooksは、Claude Codeの特定のタイミングで自動で処理を実行する仕組みです。ハンドラにはcommand（シェル実行）、http（外部APIへPOST）、prompt（LLMによる判定）、agent（サブエージェントによる検証）の4タイプがあります。今日はcommandを中心に紹介します。

3つ実践例を紹介します。例1と例2は同じタイミングの例なので、実際にはプロジェクトに合わせて選んでください。

1つ目、ファイル編集後にlintを自動修正。Claudeがファイルを編集するたびlintが走ります。

2つ目、ファイル編集後にテストを自動実行。テストが失敗するとClaudeに通知されます。Claudeはテスト結果を見て自動で修正を試みます。

3つ目、完了通知。macOSの `say` コマンドで音声通知。長い作業を投げておいて、終わったら教えてもらう使い方です。

重要なのは、Hooksが失敗するとClaudeにその結果が通知され、Claudeが自分で修正を試みてくれること。自動チェックと自己修正の組み合わせが強力です。

## Slide 14: MCPとは

**タイプ**: コンテンツ / **ラベル**: PART 2 — Hooks & MCP

**MCP = Model Context Protocol — 外部サービスとの連携プロトコル**

```
[Claude Code] ←MCP→ [Figma]     デザインからコンポーネント生成
[Claude Code] ←MCP→ [Chrome]    ブラウザ操作・E2Eデバッグ
[Claude Code] ←MCP→ [Jira]      チケット管理の自動化
```

**判断基準**:

| 状況 | 推奨 |
|---|---|
| CLIで十分（GitHub、AWS等） | CLI使用（gh、aws等） |
| 複雑な操作の連携が必要 | MCP検討 |
| ブラウザ操作が必要 | MCP使用（Chrome DevTools） |

→ **原則: CLIで足りるならCLI。MCPは「CLIで足りないとき」の選択肢**
→ 設定は `.mcp.json` をリポジトリに置くだけ。チーム共有可能

### 台本

MCPは外部サービスとの連携プロトコルです。Figmaやブラウザ、Jiraなどとの深い連携が必要な場合に使います。

判断基準はシンプルです。CLIで十分ならCLI。Anthropic公式も「ghコマンドはGitHub MCPより多くのケースで適している」と言っています。

MCPは「CLIで足りないとき」の選択肢、と覚えておいてください。設定方法は `.mcp.json` をリポジトリに置くだけです。

## Slide 16: worktree + ISSUE.md

**タイプ**: コンテンツ / **ラベル**: PART 3 — worktree

**Claude Codeにはworktree機能が組み込まれている**

```
claude --worktree feature-auth    ← 自動でworktree作成＋ブランチ切り替え
claude --worktree bugfix-payment  ← 別ターミナルで並列実行
```

各worktreeは独立したファイル状態。CLAUDE.md・Skills・Hooksの設定もすべて引き継がれる

**`/issue-worker` スキルは `--worktree` にISSUE.md生成を足す**:

```
/issue-worker 123 456 789
  ├── worktree A: issue-123  ← ISSUE.md自動生成 → Claude起動
  ├── worktree B: issue-456  ← ISSUE.md自動生成 → Claude起動
  └── worktree C: issue-789  ← ISSUE.md自動生成 → Claude起動
```

**スキルがworktree作成〜Claude起動まで自動化する**:
1. Issue情報を取得
2. worktree + ブランチを作成
3. ISSUE.mdを自動生成（Issueの内容をそのまま記載）
4. 新しいターミナルでClaude Codeを起動

**組み込みスキルもworktreeを活用する**:

| スキル | 用途 |
|---|---|
| `/batch <指示>` | 大規模変更を5〜30ユニットに分解し、各worktreeで並列実行→PR自動作成 |
| `/simplify` | 3つのレビューエージェントを並列起動し、品質・効率・再利用性を検証 |

→ カスタムスキル（`/issue-worker`）と組み込みスキル（`/batch`）を使い分ける

→ 各worktreeにはCLAUDE.md・Skills・Hooks・サブエージェントの設定がすべて引き継がれる

### 台本

Claude Codeには `--worktree` フラグが組み込まれています。`claude --worktree feature-auth` のように起動するだけで、自動的にworktreeが作成され、独立した環境でセッションが始まります。

`--worktree` だけでも並列作業はできますが、Issue番号との紐付けやISSUE.mdの自動生成まではやってくれません。PART 1で紹介した `/issue-worker` は、この部分を自動化するスキルです。Issue番号を渡すだけでworktree作成、ISSUE.md生成、Claude起動までが一気に進みます。

組み込みスキルもworktreeを活用しています。`/batch` は大規模な変更を自動で分解し、各ユニットをworktree隔離で並列実行してPRまで作ってくれます。たとえば「src/以下のログ出力をすべてstructured loggingに移行して」と指示すると、対象を調査して5〜30のユニットに分け、それぞれ独立して実行します。

`/simplify` は実装後のコードレビュー用です。3つのレビューエージェントが並列で品質・効率・再利用性をチェックし、問題があれば自動修正します。

`/issue-worker` のようなカスタムスキルと、`/batch` `/simplify` のような組み込みスキルを組み合わせることで、開発フロー全体をカバーできます。

ポイントは、`--worktree` で作られた環境にはCLAUDE.md、Skills、Hooks、サブエージェントの設定がすべて引き継がれること。PART 1とPART 2で整えた仕組みが、そのまま各並列作業で活きてきます。

## Slide 17: 大規模開発の全体像

**タイプ**: コンテンツ

**Claude Codeを強化する仕組みの全体像**

| 仕組み | 役割 | 設定場所 |
|---|---|---|
| CLAUDE.md | 常識を教える | プロジェクトルート |
| Skills | 手順を仕組み化 | `.claude/skills/` |
| Hooks | 品質チェックを自動化 | `.claude/settings.json` |
| サブエージェント | 重い作業を委譲 | `.claude/agents/` |
| MCP | 外部サービスとつなぐ | `.mcp.json` |
| worktree | 並列開発を実現 | Git機能 |

**大規模開発のフロー**:

1. **セットアップ** — CLAUDE.md / Skills / Hooks / サブエージェント / MCP を設定
2. **Issue割り当て** — `/issue-worker` でworktree + ISSUE.md を作成
3. **並列実行** — 各worktreeでClaude Codeを起動し、独立して作業を進める
4. **品質担保** — Hooksが自動でlint・テスト
5. **完了** — PR作成 → 人間がコードレビュー → マージ → worktree削除

→ すべてgit管理可能。チーム全員が同じ環境で並列開発できる

### 台本

ここまでの全体像をまとめます。

CLAUDE.mdで常識を教え、Skillsで手順を仕組み化し、Hooksで品質チェックを自動化し、サブエージェントで重い作業を委譲し、MCPで外部サービスと連携する。

そしてworktreeで、これらがすべて揃った環境を複数同時に立ち上げて並列開発する。

大規模開発のフローとしては、まずセットアップ。次にIssueごとにworktreeを作成。各worktreeでClaude Codeを起動し、独立して作業を進めます。Hooksが自動でチェックし、最後にPR作成。自動チェックはレビュー負荷を下げるためのもので、最終的なコードレビューとマージ判断は人間が行います。

実際にこの仕組みを導入しているプロジェクトでは、worktreeを19個同時に稼働させ、週に3〜5個のIssueを並列で進めています。MRは1日1〜2件のペースで出ています。worktreeの自動生成により、手作業で5〜10分かかっていた環境構築が1分以内に終わります。
